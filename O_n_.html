Графік функцій складності (логарифмічний масштаб)

Приклади алгоритмів за складністю

O(1) - Константна складність

Доступ до елемента масиву за індексом
Час доступу не залежить від розміру масиву, оскільки використовується адресна арифметика.
Вставка/видалення з початку зв'язного списку
Операція виконується за фіксований час, незалежно від розміру списку.

O(log n) - Логарифмічна складність

Бінарний пошук у відсортованому масиві
На кожному кроці пошуковий простір зменшується вдвічі.
Пошук у збалансованому бінарному дереві пошуку
Висота дерева зростає логарифмічно від кількості елементів.

O(n) - Лінійна складність

Лінійний пошук у невпорядкованому масиві
В найгіршому випадку потрібно перевірити кожен елемент.
Обхід зв'язного списку
Потрібно відвідати кожен вузол списку один раз.
Знаходження максимуму/мінімуму в масиві
Потрібно перевірити кожен елемент для визначення найбільшого/найменшого.

O(n log n) - Лінійно-логарифмічна складність

Швидке сортування (в середньому випадку)
Рекурсивно ділить масив на частини і сортує їх окремо.
Сортування злиттям
Рекурсивно ділить масив навпіл, потім зливає відсортовані частини.
Пірамідальне сортування
Побудова піраміди за O(n) і n разів вилучення кореня за O(log n).

O(n²) - Квадратична складність

Сортування бульбашкою
Для кожного елемента порівнюємо з усіма іншими елементами.
Сортування вибором
Для кожної позиції шукаємо мінімальний елемент серед решти.
Наївне множення матриць
Для кожного елемента результату обчислюємо суму добутків.

O(n³) - Кубічна складність

Множення трьох матриць
Потрібно три вкладені цикли для обчислення кожного елемента.
Алгоритм Флойда-Уоршелла для пошуку найкоротших шляхів
Три вкладені цикли для перебору всіх пар вершин через проміжні вершини.

O(2ⁿ) - Експоненційна складність

Задача комівояжера (метод повного перебору)
Перебір всіх можливих перестановок міст (n! варіантів).
Розв'язання задачі про рюкзак (метод повного перебору)
Перебір всіх підмножин предметів (2ⁿ варіантів).

O(n!) - Факторіальна складність

Задача комівояжера (деякі алгоритми)
Повний перебір усіх можливих маршрутів між n містами.
Генерація всіх перестановок множини
Кількість перестановок n-елементної множини дорівнює n!.

Теорія обчислювальної складності

Що таке обчислювальна складність? +

Обчислювальна складність — це галузь теорії обчислень, що вивчає кількість ресурсів (часу, пам'яті), необхідних для виконання алгоритмів. Вона дозволяє оцінити ефективність алгоритмів та передбачити їхню поведінку при зростанні обсягу вхідних даних.

Основним інструментом аналізу складності є асимптотичні позначення, серед яких найважливішим є "О-велике" (Big O notation).

Поняття "О-велике" (Big O Notation) +

О-велике — це математичне позначення, що описує асимптотичну поведінку функції. В інформатиці воно використовується для класифікації алгоритмів за часом їх виконання або обсягом пам'яті, що вони використовують, у залежності від розміру вхідних даних.

Формально, кажуть, що функція f(n) є O(g(n)), якщо існують такі додатні константи c та n₀, що для всіх n ≥ n₀ виконується нерівність f(n) ≤ c·g(n).

Це означає, що f(n) зростає не швидше, ніж g(n), з точністю до постійного множника.

Основні класи складності +
  • O(1) - Константна складність: Час виконання не залежить від розміру вхідних даних.
  • O(log n) - Логарифмічна складність: Час виконання зростає логарифмічно від розміру вхідних даних.
  • O(n) - Лінійна складність: Час виконання прямо пропорційний розміру вхідних даних.
  • O(n log n) - Лінійно-логарифмічна складність: Поширена для ефективних алгоритмів сортування.
  • O(n²) - Квадратична складність: Час виконання пропорційний квадрату розміру вхідних даних.
  • O(2ⁿ) - Експоненційна складність: Час виконання подвоюється з кожним додатковим елементом вхідних даних.
  • O(n!) - Факторіальна складність: Час виконання зростає факторіально від розміру входних даних.
Практичне значення аналізу складності +

Аналіз складності дозволяє:

  • Порівнювати ефективність різних алгоритмів для розв'язання однієї задачі
  • Визначати межі застосовності алгоритмів для великих обсягів даних
  • Оцінювати необхідні ресурси для виконання алгоритму
  • Визначати, чи є задача розв'язною на практиці при заданих обмеженнях

Наприклад, алгоритм зі складністю O(n³) може бути прийнятним для n=100, але стає непрактичним для n=1000, оскільки час виконання зросте в 1000 разів.
